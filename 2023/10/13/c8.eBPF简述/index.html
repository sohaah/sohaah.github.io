eBPF简述

eBPF的起源

伯克利包过滤器（Berkeley Packet Filter）是一个 Linux kernel 中用以对来自于链路层的数据包进行过滤的架构，其位于内核中的架构如下图所示：

图片引用自eBPF 入门指指南

相比起传统的数据包过滤器而言，BPF 在内核中实现了一个新的虚拟机设计，通过即时编译（Just-In-Time compilation）技术将 BPF 指令翻译为 BPF 虚拟机的字节码，可以高效地工作在基于寄存器结构的 CPU 上。（https://arttnba3.cn/2023/05/31/EBPF_0X00/#0x00-%E4%B8%80%E5%88%87%E5%BC%80%E5%A7%8B%E4%B9%8B%E5%89%8D）

Linux kernel 自 3.18 版本起提供了扩展伯克利包过滤器（extended BPF，即 eBPF），其应用范围更广，能够被应用于更多的场景，原来的 BPF 被称为 classic BPF（cBPF），且目前基本上已经被废弃，Linux 会将 cBPF 字节码转化为 eBPF 字节码再执行。而eBPF 的概念最早源自于 BSD 操作系统中的 BPF（Berkeley Packet Filter），1992 伯克利实验室的一篇论文 “The BSD Packet Filter: A New Architecture for User-level Packet Capture”。这篇论文描述了，BPF 是如何更加高效灵活地从操作系统内核中抓取网络数据包的。

（https://zhuanlan.zhihu.com/p/378766217）

eBPF Hook图片引用自ebpf.io

跟进查看eBPF Hook，类似是一种将数据进行序列化后接口的编写，不过是对kernel的数据进行交换的数据进行处理。跟进一步查看eBPF是如何进行编写和引用的。在大多数的场景下eBPF不是直接作用，但是indirectly via projects像是Cilium，bcc,或者bpftrace则是作为一个抽象的设备运行在eBPF上并且不用特定为其编写程序只需要指定相应的目标就可以在eBPF中执行。在具体实施环境中如果没有更高等级的抽象设备或者需要直接被写入，那么eBPF程序则会被Linux kernel期望作为字节码进行装载，常见的编译器就像LLVM将编译c语言的代码成为eBPF字节码，正如上文所说的进行序列化。

eBPF 编译流程图片引用自ebpf.io

eBPF的运行过程为首先将在用户空间编写相应的 BPF 字节码程序，传入内核内核通过 verifier 对字节码程序进行安全性检查，通过检查后便通过 JIT 编译运行，流程大致如下：

针对verifier，在eBPF中验证步骤确保eBPF程序可以安全运行。它验证程序是否满足几个条件以后才能进行 JIT 编译没有回向边（back edge）、环路（loop）、不可达（unreachable）指令，主要检查一下几点：

· 不能在指针之间进行比较，指针只能与标量进行加减（eBPF 中的标量值为不从指针派生的值），verifier 会追踪哪些寄存器包含指针、哪些寄存器包含标量值

· 指针运算不能离开一个 map 的“安全”边界，这意味着程序不能访问预定义的 map 外的内存，verifier 通过追踪每个寄存器值的上界与下界

· 不能将指针存储在 map 中或作为返回值，以避免将内核地址泄露到用户空间

针对JIT即是编译步骤将程序的通用字节码翻译成机器专用指令集，以优化程序的执行速度。这使得eBPF程序可以像本地编译的内核代码或作为内核模块加载的代码一样高效地运行。

针对eBPF map，EBPF的一个重要方面是共享收集的信息和存储状态的能力。为此，eBPF程序可以利用eBPF映射的概念来存储和检索各种数据结构中的数据。eBPFmap可以从eBPF程序访问，也可以通过系统调用从用户空间的应用程序访问。

返回到eBPF的调用操作中，因为eBPF不能直接调用任何内核函数，所有在eBPF程序运行中会绑定特定的eBPF功能与特定的内核版本进行运行的，比如以下功能：

· 不生成随机数

· 获取当前时间和日期

· 访问eBPF map

· 获取进程/cgroup上下文

· 操纵网络数据包和转发逻辑

进一步总结的话，那么就是eBPF的所有操作都是通过bpf的系统调用经过Helper Calls完成的。

归并到eBPF的调用方式，eBPF的调用方式以一种尾部和函数调用的方式进行组合的，这种Tail的调用方式可以使一个eBPF程序执行完后接替执行另一个eBPF程序，类似于execve()函数对于常规函数的执行的操作。

eBPF的技术状态

已经了解了eBPF的来源的运行模式，我们提出了一个问题，为什么要使用eBPF呢？有一个很形象的例子来自于网络上的类比，20年前，网页几乎都是用静态标记语言(HTML)编写的。网页基本上是一个具有能够显示它的应用程序(浏览器)的文档。纵观今天的网页，网页已经成为成熟的应用程序，基于网络的技术已经取代了绝大多数用需要编译的语言编写的应用程序。是什么促成了这一演变的呢？

简短的答案就是来自JavaScript的出现，它开启了一场巨大的改变，使得浏览器几乎变为了一个独立的操作系统。为什么会发生这样的变化？这是因为程序员不再局限于运行特定浏览器版本的用户。不是让标准机构相信需要一个新的HTML标签，而是提供必要的基本结构块，使底层浏览器的创新速度与运行在顶层的应用程序脱钩。当然，这有点过于简单化了，因为HTML也随着时间的推移而改变，并为其成功的做出了贡献，但仅靠HTML本身的演变是不够的。让我们先来看看在引入JavaScript时至关重要的几个关键方面：

· 安全性：不受信任的代码在用户的浏览器中运行。这是通过沙箱的JavaScript程序和抽象访问浏览器数据解决了这个问题。

· 持续交付：在不需要不断发布新浏览器版本的情况下，程序逻辑的演变必须成为可能。这是通过提供足以构建任意逻辑的正确的低级构建块来解决的。

· 性能：必须以最小的开销提供可编程性。

这个问题通过引入实时(JIT)编译器得到了解决。出于同样的原因，可以在eBPF中找到与上述所有内容近乎相同的对应内容。让我们回到eBPF中相信有JavaScript的例子就能很轻松的了解到eBPF在内核中相对应的作用。Linux内核的主要目的是对硬件或虚拟硬件进行抽象，并提供一致的API（系统调用），以允许应用程序运行和共享资源。为了实现这一目标，维护了广泛的子系统和层次结构来分配这些职责。每个子系统通常允许进行一定程度的配置，以满足不同用户的需求。如果无法通过配置实现所需的行为，通常需要对内核进行更改，历史上有两种选择：

· 本地支持：更改内核源代码，并让Linux内核社区相信需要进行更改。或者等待几年后，出现新的内核版本。

· 内核模块：编写内核模块定期修复它，因为每个内核版本都可能破坏它由于缺乏安全边界，但有可能损坏您的Linux内核。

通过eBPF，现在有一种新的选项，允许重新编程Linux内核的操作，而无需更改内核源代码或加载内核模块。

eBPF的开发工具链

在实际eBPF开发中有着以下不同的工具，下面我们将简单的介绍一下。

Bcc是一个框架，允许用户编写包含eBPF程序的Python程序。该框架主要针对涉及应用程序和系统分析/跟踪的用例，其中eBPF程序用于收集统计信息或生成事件，而用户空间中的对应程序则收集数据并以人类可读的形式显示它。运行Python程序将生成eBPF字节码并将其加载到内核中。

bpftrace是用于Linux eBPF的高级跟踪语言，可以在较新的Linux内核（4.x）中使用。bpftrace使用LLVM作为后端来将脚本编译成eBPF字节码，并利用BCC与Linux eBPF子系统以及现有的Linux跟踪功能进行交互：内核动态跟踪（kprobes）、用户级动态跟踪（uprobes）和tracepoints。bpftrace语言受到awk、C以及先前的跟踪工具如DTrace和SystemTap的启发。

EBPF Go库提供了一个通用的eBPF库，它分离了获取eBPF字节码的过程以及eBPF程序的加载和管理。EBPF程序通常是通过编写高级语言创建的，然后使用CLANG/LLVM编译器编译成eBPF字节码。

Libbpf库是一个基于C/C++的通用eBPF库，它帮助将从clang/LLVM编译器生成的eBPF目标文件加载到内核中，并通过为应用程序提供易于使用的库API来抽象与BPF系统调用的交互。（https://facebookmicrosites.github.io/bpf/blog/）

eBPF初始开发实践

bpf()系统调用的执行的操作是由cmd参数决定的. 每一个操作都有通过attr传递的对应参数, 这个参数是指向公用体类型bpf_attr的指针, size参数代表attr指针指向的数据长度

cmd可以是下面的值：

BPF_MAP_CREATE: 创建一个映射, 返回一个引用此此映射的文件描述符. close-on-exec标志会自动设置

BPF_MAP_LOOKUP_ELEM在指定的映射中根据key查找一个元素, 并返回他的值

BPF_MAP_UPDATE_ELEM在指定映射中创建或者更新一个元素

BPF_MAP_DELETE_ELEM在指定映射中根据key查找并删除一个元素

BFP_MAP_GET_NEXT_KEY在指定映射中根据key查找一个元素, 并返回下一个元素的key

BPF_PROG_LOAD: 验证并加载一个eBPF程序, 返回一个与此程序关联的新文件描述符. close-on-exec标志也会自动加上。

在bpf_load中，bpf_attr剩余区域按照如下设置：

· insns是struct bpf_insn指令组成的数组

· insn_cnt是insns中指令的个数

· license是许可字符串, 为了与标志为gpl_only的助手函数匹配必须设置GPL

· log_buf是一个调用者分配的缓冲区, 内核中的验证器可以在里面保存验证的log信息. 这个log信息由多行字符串组成, 目的是让程序作者明白为什么验证器认为这个程序是不安全的(相当于编译器的日志), 随着验证器的发展, 输出格式可能会改变

· log_size是log_buf的缓冲区大小, 要是缓冲区不足以保存全部的验证器日志, 那么会返回-1, 并把errno设置为ENOSPC

· log_level是验证器日志的详细级别, 00表示验证器不会提供日志, 在这种情况下log_buf必须是空指针, log_size必须是0
